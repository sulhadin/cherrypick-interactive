# Feature: Tracker Integration (Jira / Linear / ClickUp / Any)

## Problem

Changelogs generated by the tool contain plain commit messages. Teams that use issue trackers (ClickUp, Jira, Linear, etc.) want ticket IDs in the changelog to be clickable links. Currently, there's no way to do this automatically.

## Solution

A tracker-agnostic integration that:
1. Detects ticket IDs in commit messages using a user-defined regex pattern
2. Converts them to clickable links using a URL template with `{{id}}` placeholder

No API calls, no authentication — just pattern matching and URL templating.

## Config

Stored in `.cherrypickrc.json` (same file as profiles):

```json
{
  "tracker": {
    "ticket-pattern": "#([a-z0-9]+)",
    "tracker-url": "https://app.clickup.com/t/{{id}}"
  }
}
```

Or via CLI flags (advanced):

```bash
cherrypick-interactive --ticket-pattern "#([a-z0-9]+)" --tracker-url "https://app.clickup.com/t/{{id}}"
```

Or via built-in presets (beginner-friendly):

```bash
cherrypick-interactive --tracker clickup --tracker-url "https://app.clickup.com/t/{{id}}"
cherrypick-interactive --tracker jira --tracker-url "https://team.atlassian.net/browse/{{id}}"
cherrypick-interactive --tracker linear --tracker-url "https://linear.app/my-team/issue/{{id}}"
```

Presets set the `ticket-pattern` automatically. `--tracker-url` is still required since it contains team-specific values.

## Examples

### ClickUp

```
Commit:   #86c8w62wx - Upgrade apk packages (#585)
Pattern:  #([a-z0-9]+)
URL:      https://app.clickup.com/t/{{id}}

Changelog:
- [#86c8w62wx](https://app.clickup.com/t/86c8w62wx) - Upgrade apk packages (#585)
```

### Jira

```
Commit:   PROJ-123 fix login redirect
Pattern:  ([A-Z]+-\d+)
URL:      https://team.atlassian.net/browse/{{id}}

Changelog:
- [PROJ-123](https://team.atlassian.net/browse/PROJ-123) fix login redirect
```

### Linear

```
Commit:   feat: add auth [ENG-456]
Pattern:  \[([A-Z]+-\d+)\]
URL:      https://linear.app/my-team/issue/{{id}}

Changelog:
- feat: add auth [ENG-456](https://linear.app/my-team/issue/ENG-456)
```

Note: Pattern includes the brackets `\[...\]` so the full `[ENG-456]` is replaced with the markdown link, avoiding nested bracket issues.

## CLI Changes

| Flag | Type | Description |
|------|------|-------------|
| `--tracker` | string | Built-in preset: `clickup`, `jira`, `linear`. Sets `ticket-pattern` automatically. |
| `--ticket-pattern` | string | Custom regex to capture ticket ID (must have one capture group). Overrides `--tracker` preset. |
| `--tracker-url` | string | URL template with `{{id}}` placeholder (required when using tracker) |

## Implementation Steps

1. Add `--tracker`, `--ticket-pattern`, and `--tracker-url` to yargs config (under "Tracker options" group)
2. Define built-in presets:
   - `clickup`: `#([a-z0-9]+)`
   - `jira`: `([A-Z]+-\d+)`
   - `linear`: `\[([A-Z]+-\d+)\]`
3. Create `linkifyTicket(subject, pattern, urlTemplate)` function:
   - Match `pattern` against `subject`
   - If match found, replace the matched text in-place with `[matched](url)`
   - If no match, return subject as-is
3. Create `parseTrackerConfig(argv)` function:
   - Read from CLI flags or `.cherrypickrc.json` tracker section
   - Validate: if one is set, both must be set
   - Compile regex, validate it has exactly one capture group
4. Modify `buildChangelogBody()`:
   - For each commit subject, run `linkifyTicket()` before adding to changelog
5. Modify PR description builder (if applicable):
   - Same linkification for PR body

## Pattern Matching Gotcha

Be careful with patterns that could match unintended parts of the commit message. For example, ClickUp commit:

```
#86c8w62wx - Upgrade apk packages (#585)
```

Pattern `#([a-z0-9]+)` matches both `#86c8w62wx` (ticket) and `#585` (PR number). The `linkifyTicket` function should only replace the **first match** by default.

## Security: ReDoS Protection

User-supplied `--ticket-pattern` accepts arbitrary regex. A crafted pattern with catastrophic backtracking (e.g., `(a+)+$`) can hang the CLI — especially dangerous in CI where it would stall the pipeline. Before compiling the regex, validate it with the `safe-regex` package to reject patterns with super-linear worst-case behavior.

## Validation

- `--ticket-pattern` without `--tracker-url` (or vice versa) → error
- Invalid regex → clear error message
- Regex without capture group → error: "Pattern must have one capture group for the ticket ID"
- Regex fails `safe-regex` check → error: "Pattern rejected — potential catastrophic backtracking"
- After processing all commits, log summary: "X of Y commits had ticket IDs linked" — surfaces misconfigurations when 0 matches found
